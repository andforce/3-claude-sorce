# 反追踪技术方案

## 目标

目标不是“降低一些遥测”，而是尽可能做到以下两点：

1. 不让软件向开发商上报额外的行为遥测、性能指标、实验曝光、非必要联网请求。
2. 在仍然使用其第一方业务能力的前提下，尽量减少本机可被持久关联的标识与缓存。

但必须先明确一个技术边界：

**如果你继续使用软件开发商自己的在线业务，就不可能让开发商“完全不知道你的行为与数据”。**

原因不是额外遥测，而是**业务本身就是一次对开发商服务端的请求**。只要请求还发到开发商的 API，服务端至少会看到：

- 你的认证信息或账户上下文
- 你的会话请求
- 你发给模型的输入内容
- 模型返回给你的输出
- 请求时序、频率、错误、配额、账单相关信息

所以，本方案的真实目标应定义为：

**在继续正常使用业务的前提下，尽可能切断“非业务必要的追踪面”，并把本地持久身份和出网面收缩到最小。**

---

## 结论先行

如果你要一个实际可执行、风险最低、最接近“反追踪”的方案，推荐采用四层防护：

1. **运行时关闭非必要流量**：启用 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`
2. **显式关闭所有可选遥测通道**：确保不启用 `CLAUDE_CODE_ENABLE_TELEMETRY`、`ENABLE_BETA_TRACING_DETAILED`、`OTEL_LOG_USER_PROMPTS`
3. **隔离本地配置目录**：使用独立且可轮换的 `CLAUDE_CONFIG_DIR`
4. **在操作系统或网络层做域名白名单/黑名单控制**：只允许业务必需端点，阻断遥测端点

这是“尽量不被追踪，但仍正常用业务”的最强现实方案。

## 基于文档与代码确认的追踪面

以下内容均来自 `DATA_COLLECTION_TRACKING_ANALYSIS.md` 与代码核对。

### 1. 第一方事件日志

存在第一方高保真事件上报：

- 端点：`https://api.anthropic.com/api/event_logging/batch`
- 代码：`src/services/analytics/firstPartyEventLogger.ts`
- 导出器：`src/services/analytics/firstPartyEventLoggingExporter.ts`

这一路会记录大量行为事件，例如：

- API 请求成功/失败
- 工具调用成功/失败
- 配置变更
- 模式切换
- 插件与 MCP 相关操作
- OAuth 流程

代码确认点：

- `src/services/analytics/config.ts` 中，`isAnalyticsDisabled()` 在 `DISABLE_TELEMETRY` 或 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` 生效时返回禁用。
- `src/services/analytics/firstPartyEventLogger.ts` 中，1P event logging 直接依赖 `isAnalyticsDisabled()`。

### 2. Datadog 运维日志

存在 Datadog 上报：

- 端点：`https://http-intake.logs.us5.datadoghq.com/api/v2/logs`
- 代码：`src/services/analytics/datadog.ts`

虽然 Datadog 只允许部分事件名上报，但仍属于额外行为追踪面。

同样会被 `isAnalyticsDisabled()` 关闭。

### 3. GrowthBook 远程特性与实验曝光

存在 GrowthBook 远程配置与实验曝光：

- 代码：`src/services/analytics/growthbook.ts`
- 远程宿主：`https://api.anthropic.com/`

GrowthBook 不只是拿配置，它还可能携带以下用户属性进行远程评估：

- `deviceID`
- `sessionId`
- `organizationUUID`
- `accountUUID`
- `subscriptionType`
- `email`
- `platform`

并且实验曝光会写入 1P 事件流。

代码确认点：

- `src/services/analytics/growthbook.ts` 中 `isGrowthBookEnabled()` 依赖 `is1PEventLoggingEnabled()`
- 也就是说，只要 1P analytics 被禁用，GrowthBook 路径也会随之失效

### 4. BigQuery 内部 metrics

存在指标上报：

- 端点：`https://api.anthropic.com/api/claude_code/metrics`
- 代码：`src/utils/telemetry/bigqueryExporter.ts`

这一路不是产品事件，而是 metrics 信号。

代码确认点：

- `src/services/api/metricsOptOut.ts` 中，`isEssentialTrafficOnly()` 时会直接跳过组织 metrics 开关查询
- `src/utils/telemetry/instrumentation.ts` 中 BigQuery exporter 为内部指标通道

### 5. 第三方 OpenTelemetry 导出

存在可选 OTel 外发能力，但默认需要显式开启：

- 代码：`src/utils/telemetry/instrumentation.ts`
- 开关：`CLAUDE_CODE_ENABLE_TELEMETRY=1`

如果你没有主动启用，这一路通常不会开启。

### 6. Beta tracing

存在更敏感的 tracing 能力：

- 代码：`src/utils/telemetry/betaSessionTracing.ts`
- 条件：`ENABLE_BETA_TRACING_DETAILED=1` 且设置 `BETA_TRACING_ENDPOINT`

这一能力可能记录：

- system prompt
- model output
- tool 输入输出
- 新增上下文内容

因此必须确保它完全不启用。

### 7. 本地持久标识

代码确认存在稳定设备 ID：

- 代码：`src/utils/config.ts`
- 字段：`userID`
- 存储位置：全局配置目录下的配置文件
- 生成方式：随机 32 字节十六进制字符串

这不是浏览器指纹，但它是**稳定持久设备标识**。

### 8. 业务请求本身仍携带会话标识

即使关闭遥测，第一方业务请求本身仍会带会话级头：

- 代码：`src/services/api/client.ts`
- 请求头：`X-Claude-Code-Session-Id`
- 还会注入 `x-client-request-id`

这说明：

**关闭 telemetry 不等于服务端完全无法做会话级关联。**

## 威胁模型

你真正要防的是两类追踪：

### A. 非业务必要追踪

例如：

- 产品行为事件
- 实验曝光
- 运维日志
- 内部 metrics
- 非必要的远程配置拉取
- 自动更新、release notes、grove、官方 MCP 注册表等附加联网

这部分，代码里已经有现成开关，能压得很低。

### B. 业务本身带来的可观察性

例如：

- 你请求了什么模型
- 你何时请求
- 你的输入输出内容
- 认证主体是谁
- 会话请求是连续的还是高频的

这部分，只要还在用第一方在线业务，就无法消灭，只能缩减辅助标识与本地留痕。

## 推荐方案

## 第一层：使用 `essential-traffic` 模式

这是主开关，优先级最高。

设置：

- `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`

代码依据：

- `src/utils/privacyLevel.ts`

生效后会进入 `essential-traffic` 级别。代码与文档都明确说明，该级别会关闭：

- analytics
- telemetry
- auto-updates
- grove
- release notes
- model capabilities 查询
- 其它非必要网络流量

这是最重要的一步。

如果只设置 `DISABLE_TELEMETRY=1`，只能进入 `no-telemetry`，它会关闭 analytics，但不会像 `essential-traffic` 那样收紧所有非必要流量。

所以在你的目标下，**优先选 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`，不要只停留在 `DISABLE_TELEMETRY=1`。**

---

## 第二层：显式关闭所有可选遥测通道

即使已经启用 `essential-traffic`，仍建议把下面这些开关显式保持关闭，避免被外部环境或继承环境污染：

- 不设置 `CLAUDE_CODE_ENABLE_TELEMETRY`
- 不设置 `ENABLE_BETA_TRACING_DETAILED`
- 不设置 `BETA_TRACING_ENDPOINT`
- 不设置 `OTEL_LOG_USER_PROMPTS`
- 不设置 `OTEL_EXPORTER_OTLP_ENDPOINT`
- 不设置 `OTEL_METRICS_EXPORTER`
- 不设置 `OTEL_LOGS_EXPORTER`
- 不设置 `OTEL_TRACES_EXPORTER`

原因：

1. `CLAUDE_CODE_ENABLE_TELEMETRY=1` 会启用 3P OTel 导出能力，代码在 `src/utils/telemetry/instrumentation.ts`
2. `OTEL_LOG_USER_PROMPTS=1` 会让 prompt 内容不再被 `<REDACTED>`，代码在 `src/utils/telemetry/events.ts`
3. `ENABLE_BETA_TRACING_DETAILED=1` + `BETA_TRACING_ENDPOINT` 会启用更细粒度 tracing，代码在 `src/utils/telemetry/betaSessionTracing.ts`

建议策略：

- 不是“设为 false”
- 而是**完全不设置**

这样最稳妥。

---

## 第三层：隔离并轮换本地配置目录

即使网络侧遥测关闭，本地仍会产生稳定设备标识与缓存。

代码确认：

- `src/utils/envUtils.ts` 中，`getClaudeConfigHomeDir()` 优先读取 `CLAUDE_CONFIG_DIR`
- `src/utils/config.ts` 中，`userID` 持久保存在全局配置目录
- `src/services/analytics/firstPartyEventLoggingExporter.ts` 中，失败事件缓存也会写入配置目录下的 `telemetry/`

因此建议：

1. 不直接复用默认 `~/.claude`
2. 使用独立的 `CLAUDE_CONFIG_DIR`
3. 为高隐私使用场景准备单独目录
4. 需要进一步弱化长期关联时，周期性轮换该目录

这样能降低以下持久状态的长期复用：

- `userID`
- cached GrowthBook features
- metrics 状态缓存
- 各类历史缓存
- telemetry 失败重试文件

但注意：

### 轮换配置目录会带来副作用

- 需要重新登录
- 会丢失部分本地设置与缓存
- 某些体验会退化

所以推荐做法不是“每次都换”，而是：

- 日常隐私模式用一个**固定但独立**的配置目录
- 高敏场景再临时使用**一次性配置目录**

---

## 第四层：在网络层做强制收口

只依赖应用内部开关还不够，最稳妥的是在宿主机或出口网络做强制控制。

### 应阻断的端点

优先阻断以下已确认的追踪端点：

- `https://http-intake.logs.us5.datadoghq.com/api/v2/logs`
- `https://api.anthropic.com/api/event_logging/batch`
- `https://api.anthropic.com/api/claude_code/metrics`
- `https://api.anthropic.com/api/claude_code/organizations/metrics_enabled`
- `https://api.anthropic.com/mcp-registry/`

说明：

- Datadog 是额外运维日志
- `/api/event_logging/batch` 是第一方事件日志
- `/api/claude_code/metrics` 是内部 metrics
- `/metrics_enabled` 是 metrics 开关查询，本身也是额外请求
- `mcp-registry` 在 `essential-traffic` 下已会跳过，但网络层再拦一次更稳

### 对 `https://api.anthropic.com/` 不能全局一刀切

因为第一方业务也走这个域名。

所以网络层的正确策略不是“封整个域名”，而是：

- **按路径或精确域名能力做细分拦截**
- 如果你的防火墙不支持按路径拦截，就优先依赖应用内 `essential-traffic`，不要粗暴封整个 `api.anthropic.com`

否则你会把业务一起封死。

---

## 建议的最终配置基线

### 必须项

- 启用 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`
- 不启用 `CLAUDE_CODE_ENABLE_TELEMETRY`
- 不启用 `ENABLE_BETA_TRACING_DETAILED`
- 不启用 `OTEL_LOG_USER_PROMPTS`
- 使用独立 `CLAUDE_CONFIG_DIR`

### 建议项

- 同时设置 `DISABLE_AUTOUPDATER=1`

虽然 `essential-traffic` 已足以阻断 auto-update，但额外显式关闭更稳，代码依据在 `src/utils/config.ts` 的 `getAutoUpdaterDisabledReason()`

### 网络层建议

- 阻断 Datadog 端点
- 阻断 `/api/event_logging/batch`
- 阻断 `/api/claude_code/metrics`
- 阻断 `/api/claude_code/organizations/metrics_enabled`

## 可选强化方案

## 方案 A：隐私模式专用运行环境

做法：

- 为该软件单独准备一个最小化 shell 环境
- 只保留业务运行必须的认证变量
- 显式移除所有 OTel、Beta tracing、调试相关环境变量

适用：

- 你怀疑系统里已有全局环境变量污染
- 你经常在不同项目里切换，怕继承到不该有的 exporter 配置

优点：

- 风险最低
- 不需要改代码

缺点：

- 管理起来更繁琐

## 方案 B：一次性配置目录

做法：

- 每次敏感使用时都用新的 `CLAUDE_CONFIG_DIR`
- 结束后删除该目录

优点：

- 本地长期标识最弱
- 最大限度减少 `userID` 复用

缺点：

- 每次都要重新登录或重新准备认证
- 使用体验明显下降

## 方案 C：改走第三方推理后端

如果你的真实目标是：

**既要用这个客户端界面，又不想把业务内容持续交给原开发商第一方服务端观察**

那么唯一真正有效的方向不是“继续走第一方 API 但不让其看到”，而是：

- 改用 Bedrock
- 改用 Vertex
- 改用 Foundry
- 改用自定义兼容后端

代码依据：

- `src/services/analytics/config.ts` 中，Bedrock / Vertex / Foundry 会直接使 `isAnalyticsDisabled()` 返回关闭
- `src/services/api/client.ts` 中，客户端支持多种 provider

这是唯一能同时显著降低“业务内容被第一方服务端直接观察”的办法。

但这已经不是“继续完整使用第一方业务”，而是“继续使用客户端”，目标不同。

---

## 不建议的做法

### 1. 只删除本地 `userID`

这只能削弱稳定设备标识，不能阻断：

- 会话级业务请求
- OAuth 账户关联
- 服务端从业务流量推断行为

所以它不能单独成立，必须配合 `essential-traffic` 和网络层控制。

### 2. 只设置 `DISABLE_TELEMETRY=1`

这会关闭 analytics，但不如 `essential-traffic` 收得彻底。

你的目标是“尽可能不被追踪”，因此默认应直接使用更严格的 `essential-traffic`。

### 3. 粗暴封禁整个 `api.anthropic.com`

这会把业务请求一起封掉，无法正常使用。

### 4. 相信“未启用 OTel 就完全无追踪”

错误。

因为 1P event logging、Datadog、GrowthBook 不是同一条 OTel 配置链，代码里是独立路径。

---

## 残余风险

即便采用本方案，仍然存在以下不可消除的残余风险：

1. **业务可见性无法消除**  
   只要还在用第一方在线业务，服务端就能看到你的业务请求、响应、时序、账户与会话上下文。

2. **会话级关联仍然存在**  
   业务请求头里有 `X-Claude-Code-Session-Id`，代码位于 `src/services/api/client.ts`。

3. **账户级关联仍然存在**  
   只要使用 OAuth 或第一方账户体系，账户本身就是天然标识。

4. **输入内容是否被用于服务端内部处理，不由客户端完全决定**  
   客户端最多控制“额外追踪通道”，不能单方面决定服务端如何处理正常业务请求。

---

## 推荐落地顺序

建议按以下顺序实施：

1. 启用 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`
2. 确认没有启用任何 OTel / Beta tracing / prompt logging 环境变量
3. 将 `CLAUDE_CONFIG_DIR` 改为独立目录
4. 在网络层阻断已确认的遥测端点
5. 如仍不放心，再切换到一次性配置目录
6. 如果你的真实要求是“连业务内容也不想被第一方开发商看到”，则不要继续使用第一方在线业务，改走第三方 provider

---

## 最终判断

如果你的要求是：

**“我还想继续正常使用开发商自己的在线业务，但开发商不要追踪我的任何行为与数据。”**

那么从技术上讲，**这个目标不可同时满足**。

能做到的最强版本是：

**阻断所有非业务必要追踪，只保留业务本身不可避免的服务端可见性。**

在当前代码基础上，最合理、最完整、最稳妥的组合就是：

- `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`
- 不启用任何 OTel / Beta tracing / prompt logging
- 使用独立或一次性 `CLAUDE_CONFIG_DIR`
- 在网络层阻断已知追踪端点

如果你要的是“开发商连业务内容都不要看到”，那就不能继续走第一方业务通道，只能换 provider。
